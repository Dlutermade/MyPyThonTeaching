# 前言

# 寫程式之前的閒聊話

為啥要寫程式呢?

因為

這個時代我們已經把生活中從有形的過程簡化成無形的過程，而程式就是指那個無形，現實則是指那個有形，我們是透過程式在模擬現實的場景

# 寫程式都在幹嘛

# 我們通常都在寫一件事情的經過和過程，即周遭變數的變化，通常寫程式不是指寫Html和Css這種網頁碼，也就是說呢程式會分為前端和後端，前端注意畫面也就是使用者看的到的介面，後端注意程式流程細節也就是資料的調整與改變

# 學會程式的好處

# 學會到學好的分析

# 1. 可以把生活周遭一些事情透過程式自動化解決，亦可不用都勞累於自己

# 2. 程式就是一個通用計算機，可以計算任何數學作業，即不用再慢慢算數

# 3. 大學可以放鬆，資訊課程輕鬆躺過 (作者高中程式當躺著過)

# 4. 很多普通大學特殊選材招生，可以去報名，前題是要有豐厚的程式競賽和檢定實力 EX: APCS、TOI、IOI、NPSC...等等(注意像是TQC類型的程式證照基本上沒作用)

TQC證照，題目比較單調，且過於簡單，考的是基礎語法為主，普通大學需要的豐厚程式基礎理論底子...等等，並非是要一個只會單一語言且不深入的玩家。

# 5. 科大招生可以成為亮點，職高能專精程式的人少之又少，普遍來說都是普通科比較強，因為他們從國小五、六年級階段就開始培養，亦可說我們早期受到的資源就少，也可以靠天分進行補足。

# 最關鍵地方，台灣每年會派4名高中生去比國際資訊奧林匹亞，成績不錯可以保送麻省理工...等等，而進入到台灣資訊奧林匹亞選手培訓營第二階段(每年大約10個)，可以<直接>保送台、清、交等電機系或資工系方面的


# 那些人學程式比較快

# 以下採影響排名

# 1. 推理能力

因為: 因為寫程式就是在從推理事情的經過即生活周遭變數的變化

EX: 明明愛排列 先創10個串列或陣列，分別代表個位數，再根據個位數的陣列進行排序達到個位數一樣卻由大到小排列，然後再把個位數由較小到大輸出

# 2. 想像力

因為: 有時候你可以看到有些人在寫程式手在那邊動來動去等等，我們可以發現他們是在想像資料的移動變化等等，這種想像力並非<天馬行空>這很重要!!，而是有建設性的想像，透過先前已知條件進行想像下一步會是怎樣

# 3. 數量視覺化

因為: 我們寫一些演算法或是解題都是在推算數量是怎某減少的，有些人無法推理，但可以透過量化視覺化概念呈現

EX:下面圖片 M為總數量，N為總腳量

# 4. 數理能力

先強調不是算術能力，是對數字的感覺，像是作者對於程式考卷看個五秒左右就可以看出圈數公式，但作者數學成績總是不高在平均左右，因為公式沒記好，或是題目理解錯誤，以及計算失誤

因為: 假設是偏普通科在考的程式EX:NPSC、(全國、北區、高雄)資訊學科能力競賽、TOI(台灣資訊奧林匹亞)...這類有不少題目都是靠數論推理公式的，以職科觀點並不會特別出這方面類型題目

# 5. 英文能力

先強調作者英文很爛，沒及格過依然可以程式校內第二，會五種程式語言以上，別因為英文而放棄程式

因為: 程式語法很多，雖然都是極短的單字可能最長也才7~10個字長左右爾以，普遍會用到的單字大約都在4個字長左右，英文能力好，可以比較理解為啥這個過程是用這個單字當作保留字

EX: While、For、In、Int、Float、Else、Elif、If、Def、Return、True、False、Bool、String、List、Set、Dict、Tuple...等等

# 重點: 數學和英文絕對不是影響你學程式的關鍵，作者數學和英文也很爛的


# 程式的兩大走向

# 專案開發型

重視點

在程式開發上有一個很專業的概念叫做SOLID原則我們現在只舉例其中的S

  1. Single responsibility principle 單一功能原則 => 即不要把太多事情都放在同一個地方做，應該要拆開來即一個函式來判斷是否是質數，另一個函數判斷b和a是否為因數關係，兩者結合就是質因數。
  2. 重視代碼抽象化，意思是指可以更加重複利用的代碼，達到減少多餘代碼
  3. 重視命名規則，因為通常專案是一個Team在維護並非一個人寫

現代工程師都比較流行camel case命名法即第一個單字小寫之後單字首字大寫 => studentCount、isWin

早期則是使用snake_case即單字都小寫但用_作為串接名稱 => student_count 、is_win

作者是比較流行用Pascal case命名法即單字首字都大寫 => StudentCount、IsWin

警告

不要使用_add或是__add這種底線在前面的命名方式，因為在Python世界中這具有特別意思

# 程式競賽型

重視點

  1. 迴圈圈數減少，可能有些人寫A題目要跑n^3可是在專業選手的思維下可以縮減到N * Log2(n)
  2. 程式時間，同上圈數越少則時間也會減少，我們會在乎程式時間是因為資料量n大迴圈會很多圈
  3. 程式命名方式很隨便，通常為了縮短時間，比如高度打成h，而總高度打成H，第一個輸入值a，第二個輸入值b，前兩個B2(B是指before)
  4. 秀技以及一些超乎常人思維的炫泡寫法，達到更快速度寫完程式，程式競賽通常會比時間

# 學程式的態度

  1. 絕對不要抄，程式只會越抄越爛，要抄也是可以前提是在抄的情況下，去理解他的操作方式，思考模式

很多演算法、資料結構的書都是這樣的

  1. 不會就問，不要只會看旁邊同學怎某寫就照打

  2. 程式是靠理解，不是背的

  3. 同樣的事情，可以呈現二十多種寫法，這就是程式的精隨，同樣的事情作法不同

排序法就是如此

  1. 遇到問題先想半小時到一小時想不出來，才去看解答並嘗試閱讀他是怎某解的
